# Hierarchy Rules

Hierarchy ensures the user instantly knows what to look at, what to read, and what to click.

## 1. The Squint Test
- If you squint at the interface, the blurred shapes should clearly indicate what the most important element on the screen is. If it's a muddy gray wash, the hierarchy has failed.

## 2. One Primary Action Per View
- There should be exactly one visually dominant Call to Action (CTA) on the screen. 
- All other buttons must be visually distinct and secondary (e.g., ghost buttons, outlined buttons, or simple text links).

## 3. Size ≠ Importance
- You do not always need to make things bigger to make them important.
- A small, brightly colored badge surrounded by ample whitespace carries more visual weight than a large, low-contrast header packed tightly with text.

## 4. The Z-Axis Hierarchy
- Use elevation to indicate interactivity and state.
- Background: The canvas.
- Layer 1: Cards and structural containers.
- Layer 2: Floating elements, sticky navs (with backdrop blur).
- Layer 3: Modals, command palettes, toasts.
- Ensure shadows and borders reflect this physical stacking order.

## 5. De-emphasize to Emphasize
- When a designer wants to emphasize an element, the amateur instinct is to make it louder (bigger, brighter).
- The elite approach is to make everything else *quieter*. Reduce the opacity of surrounding text to make the core data point pop.
